# WebTemplate Architecture

A modular Django + React + Bulma scaffolding for internal web applications. Modules (e.g. `userauth`, `helloworld`) are self-contained units that add functionality by being dropped into the project and installed.

---

## Design Principles

- **Isolation first.** Backend and frontend code within a module are independent of each other. Neither should import directly from the other's layer.
- **Core is the only glue.** Inter-module dependencies are declared in `module.json` and resolved by `core`. Modules never import from each other directly.
- **Convention over configuration.** `install.py` manages everything — symlinks, manifests, migrations. Follow the conventions and it works.
- **Each module is a git repo.** Modules live as git submodules under `modules/`. This makes versioning, sharing, and swapping modules straightforward.
  (this has not be done yet but is the plan)

### Symlinks

`install.py add` creates a symlink under `core/backend/<name>` pointing to the module's Django app directory, making the app importable as a top-level package without `sys.path` hacks.

- `core/backend/<name>` → `../../modules/<name>/backend/<name>`

The frontend uses npm workspaces directly — no symlinks needed. The workspace glob `../../modules/*/frontend` in `core/frontend/package.json` discovers all module frontend packages automatically.

All installed modules have a `core/backend/<name>` junction/symlink. The `installed_modules.py` generated file contains no `sys.path` manipulation — app discovery relies solely on the junctions and Django's standard project-root path.

Symlinks (where they exist) are tracked in git as text files containing the target path. `git clone` preserves them on Linux/macOS.

## Module Manifest (`module.json`)

Declares everything `install.py` needs to know.

```json
{
  "name": "auth",
  "version": "1.0.0",
  "django_app": "auth",
  "npm_package": "@modules/auth",
  "url_prefix": "auth",
  "requires": [],
  "pip_packages": [
    "djangorestframework-simplejwt>=5.3"
  ],
  "django_apps": [
    "rest_framework",
    "allauth"
  ],
  "middleware": [
    "allauth.account.middleware.AccountMiddleware"
  ],
  "settings_defaults": {
    "ACCOUNT_EMAIL_VERIFICATION": "optional"
  },
  "description": "JWT authentication, user management, and permission groups."
}
```

`requires` lists other module names that must be installed first. `install.py` checks this before proceeding.

`pip_packages` lists Python packages (pip specifiers) that `install.py` will install into the venv. If omitted, `install.py` falls back to `modules/<name>/backend/requirements.txt`.

`django_apps` lists third-party Django app labels added to `INSTALLED_APPS` (deduplicated across modules).

`middleware` lists middleware strings appended to `MIDDLEWARE` (deduplicated across modules).

`settings_defaults` provides key/value settings merged into Django settings at startup via `MODULE_SETTINGS`. Merge semantics:

- **Dict values** are deep-merged into any same-named setting already defined in `base.py` or by an earlier-installed module. A module can therefore add a key to `REST_FRAMEWORK` without replacing the whole dict.
- **List values** are appended to any same-named list.
- **Scalar values** overwrite any same-named setting; `install.py` warns when two modules set the same scalar key. The last module in topological dependency order wins (dependents override their dependencies).
- **Callables and lazy objects** cannot come from `module.json` (JSON limitation) — keep those in `core/backend/core/settings/base.py`.

## How Modules Are Registered

### Backend

`install.py` writes `core/backend/core/installed_modules.py` (generated, gitignored — never commit it) with four exports:

- `INSTALLED_MODULE_APPS` — the module Django app labels, topologically sorted.
- `MODULE_EXTRA_APPS` — third-party Django apps declared via `module.json` `django_apps`.
- `MODULE_EXTRA_MIDDLEWARE` — middleware strings declared via `module.json` `middleware`.
- `MODULE_SETTINGS` — settings from `module.json` `settings_defaults`. Dicts are deep-merged into existing same-named settings; lists are appended; scalars overwrite (last in topological dependency order wins — dependents override their dependencies).

`core/backend/core/settings/base.py` imports these and extends `INSTALLED_APPS`, `MIDDLEWARE`, and globals at startup. No manual edits to settings are needed for module configuration.

`core/urls.py` reads `INSTALLED_MODULE_APPS` at startup and mounts each module's `urls.py` at `/api/<module_name>/`.

### Frontend

`install.py` regenerates `core/frontend/src/modules.js` after any module change. It collects four arrays from every installed module:

- `moduleRoutes` — React Router route definitions, merged into `App.jsx`.
- `moduleNavItems` — Navbar links.
- `moduleProviders` — Provider components (e.g. `AuthProvider`, `GoogleProvider`) wrapped around the app tree in `main.jsx` via `ModuleProviders`.
- `moduleNavbarEnd` — Components rendered in the right slot of the Navbar.

`modules.js` is a generated file — never edit it by hand. It should be committed after any install/uninstall so the repo reflects the current module set.

## Install Workflow

```bash
# Install a module
python install.py add auth

# Remove a module
python install.py remove auth

# Regenerate manifests after manual changes
python install.py regen
```

`install.py` detects circular dependencies in the `requires` graph and aborts with an error before making any changes.

`install.py add` does the following in order:

1. Reads `modules/<name>/module.json`, checks `requires`
2. Runs `pip install` for `pip_packages`; falls back to `modules/<name>/backend/requirements.txt` if `pip_packages` is absent
3. Creates `core/backend/<name>` symlink → `../../modules/<name>/backend/<name>`
4. Regenerates `core/frontend/src/modules.js` and `core/backend/core/installed_modules.py`
5. Runs `npm install` from `core/frontend/` (workspace picks up the new package)
6. Runs `python core/backend/manage.py migrate`

`install.py remove` regenerates both manifests (excluding the module), removes the `core/backend/<name>` symlink, and prints the exact `migrate <app_label> zero` command to roll back the module's database schema. Pass `--run-migrations` to execute the rollback automatically (migrations run first, while the app is still discoverable, before the symlink is removed):

```
python install.py remove --run-migrations <module_name>
```

If the module has no migration files the rollback step is skipped silently.

## Development Quickstart

After cloning:

```bash
git submodule update --init --recursive           # initialise all module submodules
python install.py regen                           # regenerate installed_modules.py and modules.js
pip install -r core/backend/requirements.txt
npm install --prefix core/frontend
python core/backend/manage.py migrate
```

To run (two terminals):

```bash
python core/backend/manage.py runserver           # terminal 1 — Django on :8000
npm run dev --prefix core/frontend                # terminal 2 — Vite on :5173
```

Vite proxies `/api/*` to Django in dev mode (configured in `vite.config.js`).

## Inter-Module Communication

Modules must not import from each other. Cross-module communication uses:

- **Django signals** — two cases, handled differently:
  - *Intra-module*: a signal emitted and consumed entirely within one module. Define the signal and connect its handlers in that module's `signals.py`; register in `AppConfig.ready()`. No cross-module import.
  - *Cross-module*: signal definitions that another module must listen to go in `core/core_app/signals.py`. Both the emitting module and the listening module import only from `core`. Never define a signal inside module A that module B imports directly — that is a cross-module import and violates the isolation rule.
- **REST API** — frontend modules call each other's API endpoints, never import JS across module boundaries.
- **Shared models from `core`** — the `core` Django app owns the `User` model and any other primitives shared across modules. Modules import from `core`, not from each other.

## Frontend Conventions

- All API calls for a module live in that module's `api.js`. No fetch/axios calls in components.
- Routes are declared with whatever paths make sense — there is no enforced module-name prefix. `/login`, `/signup`, `/billing/portal` are all valid. Keep paths flat and descriptive. **Path uniqueness across modules is the module author's responsibility; `install.py` enforces it** by scanning each module's `routes.jsx` / `routes.js` and aborting on exact duplicates. Parameterized routes with the same structure (e.g. `/user/:id` and `/user/:name` from different modules) produce a warning rather than an error. **Known limitation:** detection is regex-based and only catches literal string `path` values — paths defined via variables, computed expressions, or spread operators will not be detected.
- Bulma classes are used directly. No additional CSS framework. Module-specific styles go in a `styles.css` co-located with the component.
- No global state library is assumed. Modules export React contexts (e.g. `AuthProvider` from `userauth`) via the `providers` array in `index.js`, which `main.jsx` wraps around the app tree. `core` owns `CoreProvider` for site-wide config only. **Provider ordering:** providers are collected in alphabetical order by module folder name — this is not dependency-aware. If module B's provider needs context supplied by module A's provider, module A's folder name must sort before module B's alphabetically (e.g. `auth` before `billing`). Cross-module provider dependencies should be rare; prefer signals or the API instead.

---

## Backend Conventions

- Each module's `urls.py` uses a consistent namespace matching the module name.
- All views are class-based (DRF `APIView` or `ViewSet`).
- Models use `core.models.User` (via `settings.AUTH_USER_MODEL`) for any user FK — never Django's `auth.User` directly.
- Migrations are committed to the module repo and must not depend on migrations in other modules.
- `AppConfig.ready()` is the only place signal handlers are connected.
- Always run Django commands as `python core/backend/manage.py <cmd>` from the repo root.

## Development vs Production

| Concern | Development | Production |
|---|---|---|
| Error handling | `logging` to console / file | Sentry |
| Django debug | `DEBUG=True`, sqlite | `DEBUG=False`, PostgreSQL |
| React | Vite dev server (port 5173) | Built and served as static via Django |
| Auth secrets | `.env` file (not committed) | Environment variables |

## Docker / Deployment Notes

- **Symlinks in Docker**: use `COPY --follow-symlinks` (Docker 23+) or mount the whole repo as a volume in development. For production builds, run `install.py` inside the image before the COPY step, or create symlinks explicitly in the Dockerfile.
- **Windows**: symlinks require Developer Mode or admin rights. Not a concern for Linux deployment.
- **Git submodules**: if a module is later extracted to its own repo and added as a submodule at `modules/<name>/`, the symlinks in `core/backend/` continue to work without change.

## Rules Summary

1. **Never edit `modules.js` or `installed_modules.py` by hand.** Both are generated by `install.py`.
2. **`installed_modules.py` is gitignored.** Run `python install.py regen` after cloning or changing modules.
3. **Modules never import from each other.** Use signals or the API.
4. **All inter-module dependencies are declared in `module.json`.** If it's not declared, it won't be enforced.
5. **Module folder name, Django app label, and `module.json` `name` field must all match.**
6. **All API calls live in `api.js`.** No inline fetch in components.
7. **Migrations are self-contained.** A module's migrations must run cleanly with only `core` present.
8. **`core` owns shared models.** No module defines a model that another module's migration depends on.
9. **`.env` is never committed.** `.env.example` is always kept current.
10. **Each module is a git submodule.** Pin versions; do not use floating HEAD in production.
11. **Run `install.py` for all module changes.** Do not manually pip-install or npm-install module packages.
